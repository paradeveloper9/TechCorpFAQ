# FAQ Assistant

REST API сервис для ответов на вопросы по базе знаний с использованием RAG (в переводе с английского rag - тряпка, отрепье, тряпичный утиль, шумное веселье, крепкий известняк).

---

## Архитектурные решения

### Гексагональная архитектура (порты и адаптеры)

Проект строго следует гексагональной архитектуре: доменный слой не знает об инфраструктуре и не зависит от неё.

```
src/
├── domain/            # ядро: модели, порты, сервис
│   ├── models.py      # KnowledgeArticle, SearchResult, RAGResponse, Confidence
│   ├── ports/         # абстрактные интерфейсы (Protocol)
│   │   ├── embedding.py
│   │   ├── knowledge_base.py
│   │   ├── llm.py
│   │   └── vector_store.py
│   └── services/
│       └── rag_service.py   # единственный сервис-оркестратор
├── adapters/
│   ├── inbound/http/        # FastAPI роутер + Pydantic-схемы запроса/ответа
│   └── outbound/            # конкретные реализации портов
│       ├── embedding/openrouter.py
│       ├── knowledge_base/json_file.py
│       ├── llm/openai.py
│       └── vector_store/numpy_in_memory.py
└── infrastructure/
    ├── config.py      # Pydantic Settings, читает .env
    └── container.py   # лёгкий DI-контейнер, точка сборки
```

Зависимости направлены строго внутрь: `adapters` → `domain`, `infrastructure` → `domain`. Доменный слой не импортирует ничего из adapters или infrastructure.

### RAG-пайплайн

1. **Индексация** (при старте приложения): все статьи базы знаний эмбеддятся и сохраняются в векторное хранилище (in-memory).
2. **Ответ на вопрос**:
   - Вопрос пользователя эмбеддится.
   - В векторном хранилище выполняется поиск ближайших `top_k` соседей (по косинусному сходству).
   - Рассчитывается уровень уверенности по максимальному score среди результатов.
   - Формируется промпт из найденных фрагментов, к системному промпту опционально добавляется предупреждение о низком качестве ретривала для избежания ложных ответов.
   - LLM генерирует итоговый ответ.

### Внешние API

- **LLM**: любой OpenAI-совместимый эндпоинт (по умолчанию — OpenRouter, `openai/gpt-5.4-nano`).
- **Embeddings**: OpenRouter (`openai/text-embedding-3-small`).

**Важно**: т.к. я сам пользуюсь Openrouter, он указан и для LLM и для Embeddings. В базе своей LLM-адаптер предназначен для OpenAI API, но у Openrouter тот же формат.

### Векторное хранилище

Полностью in-memory реализация на NumPy. Не требует внешних зависимостей, но **не сохраняет состояние** между перезапусками. Индексация выполняется каждый раз при старте.

### Наблюдаемость

Адаптер `observability.py` оборачивает вызовы LLM логированием: фиксируются размер промпта, задержка и статус вызова.

## Запуск

### Предварительные требования

- Python 3.11+
- Ключи API: OpenRouter (или совместимый провайдер)

### Docker Compose

```bash
# 1. Настроить переменные окружения
cp .env.example .env
# Заполнить OPENAI_API_KEY и OPENROUTER_API_KEY в .env

# 2. Запустить
docker compose up --build
```

Сервис поднимается на `http://localhost:8000`.

### Переменные окружения

| Переменная                    | Обязательная | По умолчанию                          | Описание                                      |
|-------------------------------|:---:|---------------------------------------|-----------------------------------------------|
| `OPENAI_API_KEY`              | да  | —                                     | Ключ для LLM (OpenRouter или OpenAI)          |
| `OPENAI_MODEL`                | нет | `openai/gpt-5.4-nano`                 | Идентификатор модели                          |
| `OPENAI_BASE_URL`             | нет | `https://openrouter.ai/api/v1`        | Базовый URL LLM API                           |
| `OPENROUTER_API_KEY`          | да  | —                                     | Ключ для эмбеддинг-модели                     |
| `OPENROUTER_EMBEDDING_MODEL`  | нет | `openai/text-embedding-3-small`       | Модель эмбеддингов                            |
| `OPENROUTER_HTTP_REFERER`     | нет | `""`                                  | HTTP-заголовок Referer для OpenRouter         |
| `OPENROUTER_X_TITLE`          | нет | `RAGAssistance`                       | Заголовок X-Title для OpenRouter              |
| `OPENROUTER_TIMEOUT_SECONDS`  | нет | `30.0`                                | Таймаут запросов к OpenRouter                 |
| `KNOWLEDGE_BASE_PATH`         | нет | `data/knowledge_base.json`            | Путь к JSON-файлу базы знаний                 |
| `RAG_DEFAULT_TOP_K`           | нет | `5`                                   | Количество извлекаемых фрагментов по умолчанию|

### Запуск тестов

```bash
# Запустить все тесты
pytest

# С выводом покрытия
pytest --tb=short -v
```

## Примеры использования API

### `POST /api/ask` — задать вопрос

**Запрос:**

```bash
curl -X POST http://localhost:8000/api/ask \
  -H "Content-Type: application/json" \
  -d '{"question": "Сколько дней отпуска положено?"}'
```

**Ответ:**

```json
{
  "answer": "Сотрудникам положено 28 календарных дней оплачиваемого отпуска в год. Для определённых позиций с ненормированным рабочим днём предусмотрено дополнительно 3 дня.",
  "sources": ["kb_001", "kb_002"],
  "confidence": "HIGH"
}
```

---

**Запрос с вопросом вне базы знаний:**

```bash
curl -X POST http://localhost:8000/api/ask \
  -H "Content-Type: application/json" \
  -d '{"question": "Как настроить VPN?"}'
```

**Ответ:**

```json
{
  "answer": "В базе знаний нет информации по данному вопросу. Рекомендую обратиться в IT-поддержку.",
  "sources": ["kb_005", "kb_010"],
  "confidence": "LOW"
}
```

## Формат базы знаний

База знаний — JSON-массив статей:

```json
[
  {
    "id": "kb_001",
    "category": "vacation",
    "question": "Сколько дней отпуска положено?",
    "answer": "28 календарных дней оплачиваемого отпуска в год."
  }
]
```

## Принятые допущения

1. **Статичная база знаний.** Индексация выполняется один раз при старте. Горячее обновление базы знаний без перезапуска не предусмотрено.

2. **Единый провайдер для LLM и эмбеддингов.** Предполагается использование OpenRouter, который предоставляет единый OpenAI-совместимый API для обоих типов запросов. Переменная `OPENAI_API_KEY` передаётся в LLM-клиент, `OPENROUTER_API_KEY` — в клиент эмбеддингов; на практике это может быть один и тот же ключ.

3. **Отсутствие аутентификации и авторизации.** API не защищено. Предполагается, что защита осуществляется на уровне сетевой инфраструктуры (API Gateway, VPN и т.п.).

4. **Язык ответов определяется LLM.** Сервис не форсирует язык ответа — LLM отвечает в соответствии с содержимым базы знаний и вопросом пользователя.

5. **Пороги уверенности — эмпирические.** Значения 0.82 / 0.65 / 0.45 взяты с потолка и не откалиброваны.

6. **Отсутствие rate limiting и кэширования.** Каждый запрос выполняет полный цикл эмбеддинг + поиск + LLM-генерация. Кэширование идентичных вопросов не реализовано.
